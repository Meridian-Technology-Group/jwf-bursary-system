## Context

Noticed during the v1.0.0 cutover. The `package.json` version was bumped by
hand (`0.1.0` → `1.0.0`) in a release PR, and the git tag + GitHub release
were created manually after promotion. There is no convention or tooling that
decides *which* part of the version to bump, so the major-vs-minor-vs-patch
call is an ad-hoc, easily-forgotten human decision made at deploy time.

## Why it matters

As the cadence picks up, manual versioning drifts: tags get skipped, the
`package.json` version goes stale relative to what's actually on `main`, and
"what version is prod running?" stops having a reliable answer. The bump level
is also a *pre-deployment decision* — it should be made deliberately (does this
promotion contain breaking changes, features, or just fixes?) rather than
defaulted.

## Proposed approach

Pick one of:

1. **Conventional-commits driven (automated).** Adopt `semantic-release` or
   `changesets`. The commit history already follows Conventional Commits
   (`feat:`/`fix:`/`chore:`), so the bump level (major/minor/patch) can be
   derived automatically at promotion time, with the tag + GitHub release
   generated by CI on push to `main`. Lowest ongoing effort; requires
   discipline that `feat!:`/`BREAKING CHANGE` is used for majors.
2. **Scripted manual gate.** An `npm run release` script that prompts for the
   bump level, updates `package.json`, creates the tag, and drafts the release
   notes — keeping the human decision but removing the footguns.

Either way, define the policy explicitly: what constitutes a major vs minor vs
patch for this app, and where the decision is recorded.

## Out of scope

The first manual `v1.0.0` release itself (already shipped). This entry is about
making *subsequent* releases repeatable.
