## 1. Background

The system already ships a complete branded invitation flow for **staff users** (ADMIN / ASSESSOR / VIEWER), built around the `StaffInvitation` model, a single-use cryptographic token, a server-component acceptance shell at `/register/staff`, and a hardened `inviteStaffAction` that creates the Supabase auth user up front and rolls it back on failure.

A separate **applicant** invitation flow also exists (`Invitation` model, `createInvitationAction`, `/register?invitationId=…`), but it predates the hardened staff pattern and has several gaps. This document captures the work needed to bring the applicant flow to parity with the staff flow.

### Gaps vs the staff flow

| Concern | Staff (good) | Applicant (current) |
|---|---|---|
| URL credential | Crypto `token` (32 bytes, base64url) | Raw UUID `invitation.id` |
| Auth-user rollback on DB failure | Yes — `supabase.auth.admin.deleteUser` on rollback | No — failed invites leave orphan `auth.users` rows |
| `authUserId` captured at invite time | Yes (column on `StaffInvitation`) | No — looked up later via `listUsers()` paged scan |
| Accept audit log | Yes (`ACCEPT_STAFF_INVITATION`) | Missing |
| Name fields | Separate `firstName` / `lastName` | Single `applicantName` string, split on whitespace |
| Acceptance page | Dedicated SC shell with server-side token validation | Mixed token + invitationId page with client-side validation |
| Admin Resend / Revoke UI | n/a | Missing |

### PRD pins
- **AP-01 / UM-01 / AC-02** — invitation-only registration via unique time-limited link; email pre-filled; admin specifies email + applicant name + child name + school + round.
- **US-A01** — link expires after a configurable period (current default: 30 days).
- **EN-05** — re-assessment invite reuses the same mechanism.
- **GD-05** — token must be unguessable (crypto-random, not a UUID).

---

## 3. Decisions

| Decision | Choice | Rationale |
|---|---|---|
| URL cutover | **Hard cutover to `?token=…`** | Clean code path. In-flight `?invitationId=…` links in mailboxes stop working immediately. Backfill mitigates this — see §4.1. |
| Admin Resend / Revoke | **In scope for this PR** | Matches PRD AC-02 / EN-01 expectations. Avoids manual SQL clean-up of stuck invitations. |
| Auto-expire job (cron) | **Out of scope** | Also missing from staff flow. Defer to a later PR. |
| Replace `applicantName` with `firstName + lastName` everywhere | **Partial** | Add new fields; keep `applicantName` for now to avoid churn in seeded templates and admin list display. |
| Dropping the legacy `registerWithInvitationAction(invitationId, …)` | **Remove in this PR** | No callers remain after the URL cutover. |

## 4. Implementation

### 4.1 Schema migration

**New migration**: `prisma/migrations/<timestamp>_add_applicant_invitation_token/migration.sql`

```sql
-- 1. Add new columns
ALTER TABLE "invitations"
  ADD COLUMN "token"      text,
  ADD COLUMN "first_name" text,
  ADD COLUMN "last_name"  text;

-- 2. Backfill token on any existing PENDING rows so the NOT NULL flip is safe.
--    encode(gen_random_bytes(32), 'base64') is hex-clean and won't collide with
--    base64url tokens generated by the app.
UPDATE "invitations"
SET "token" = replace(replace(encode(gen_random_bytes(32), 'base64'), '+', '-'), '/', '_')
WHERE "token" IS NULL;

-- 3. Lock token NOT NULL + unique
ALTER TABLE "invitations" ALTER COLUMN "token" SET NOT NULL;
CREATE UNIQUE INDEX "invitations_token_key" ON "invitations" ("token");
CREATE INDEX "invitations_email_idx" ON "invitations" ("email");
```

**Prisma schema changes** (`prisma/schema.prisma`):

```prisma
model Invitation {
  // ... existing fields ...
  token       String   @unique
  firstName   String?  @map("first_name")
  lastName    String?  @map("last_name")
  // ... existing relations ...

  @@index([token])
  @@index([email])
}
```

`applicantName` stays — the seeded `INVITATION` template uses `{{applicant_name}}` and the admin list table displays it. We can compute it from `firstName + lastName` when only the structured fields are supplied.

No new RLS work needed — existing policies from `20260513090020_enable_row_level_security` already cover the table; new columns inherit the existing policy.

### 4.3 Admin invite action — hardened

**File**: `src/app/(admin)/invitations/actions.ts`

Refactor `createInvitationAction` to mirror `inviteStaffAction` (`src/app/(admin)/users/actions.ts:49`):

```text
1. Validate input (zod) — same shape, but add firstName / lastName.
2. Create Supabase auth user up front:
     supabase.auth.admin.createUser({
       email,
       password: randomBytes(24).toString("base64url"),  // throwaway
       email_confirm: true,                              // suppress OTP
       app_metadata: { role: "APPLICANT" },
     })
   Capture authUserId.
3. In one withAdminContext(tx) block:
     - Upsert Profile (id = authUserId, role = APPLICANT, firstName, lastName)
     - createInvitation(tx, { ..., authUserId, firstName, lastName })
     - createAuditLog(tx, { action: "CREATE_INVITATION", entityType: "Invitation", entityId, metadata })
4. On any DB failure → supabase.auth.admin.deleteUser(authUserId) rollback.
5. Send branded INVITATION email via Resend (template body unchanged):
     registration_link: `${appUrl}/register?token=${invitation.token}`
6. revalidatePath("/invitations").
```

The same hardening is applied to:
- `batchReassessmentInviteAction` (same file) — already iterates over holders; add the auth-user creation + rollback per row.
- `createInternalRequestAction` (`src/app/(admin)/queue/actions.ts:55`) — already creates the Supabase user via `inviteUserByEmail`; replace with the silent `createUser` + token write so the invitation has the same shape.

### 4.5 Acceptance flow — new server-component shell

**File**: `src/app/(auth)/register/page.tsx`

Replace the existing client-only page with a Server Component shell that mirrors `src/app/(auth)/register/staff/page.tsx:26`:

```text
export default async function RegisterPage({ searchParams }) {
  const token = pickString(searchParams.token);
  const tokenHash = pickString(searchParams.token_hash);

  if (tokenHash) {
    // Existing Supabase magic-link branch — render <TokenRegistration /> client.
    return <TokenRegistration tokenHash={tokenHash} />;
  }

  if (!token) {
    return <InvalidInvitation message="No invitation token was provided." />;
  }

  const validation = await validateApplicantInvitationAction(token);
  if (!validation.success) {
    return <InvalidInvitation message={validation.error} />;
  }

  return (
    <ApplicantRegisterForm
      token={token}
      email={validation.email}
      firstName={validation.firstName ?? ""}
      lastName={validation.lastName ?? ""}
    />
  );
}
```

**Remove** the `?invitationId=` branch entirely (hard cutover).

**New client component**: `src/app/(auth)/register/applicant-register-form.tsx` — direct copy of `staff-register-form.tsx` adapted to the applicant accept action. No role display; otherwise the form is identical (email read-only, firstName/lastName editable, password + confirm, HIBP validation).

### 4.6 Acceptance actions

**File**: `src/app/(auth)/register/actions.ts`

- **Remove** `validateInvitationAction(invitationId)` and `registerWithInvitationAction({ invitationId, … })`.
- **Add** `validateApplicantInvitationAction(token)` — mirrors `validateStaffInvitationAction`. Returns email, firstName, lastName, plus the existing applicant-specific fields (childName, school).
- **Add** `acceptApplicantInvitationAction({ token, firstName, lastName, password })`:
  1. Re-validate token (PENDING + unexpired).
  2. Password policy check (`validatePasswordStrength` — 12 chars + HIBP, fails open on network).
  3. `supabase.auth.admin.updateUserById(invitation.authUserId, { password, email_confirm: true })`. No `listUsers()` paged scan.
  4. In one `withAdminContext` tx:
     - Update Profile firstName / lastName.
     - If invitation has `school + childName + roundId`: create the Application row (existing logic, lifted out of `registerWithInvitationAction`).
     - `markInvitationAccepted(tx, invitation.id, invitation.authUserId)`.
     - Audit log `ACCEPT_INVITATION`.
- **Keep** `createProfileAction` for the Supabase magic-link branch.

### 4.7 Admin UX — Resend / Revoke

**File**: `src/app/(admin)/invitations/page.tsx` + `src/components/admin/invitation-row-actions.tsx` (new)

Per-row actions on each PENDING invitation in the history table:

- **Resend** — generates a fresh `token`, resets `expiresAt = now + 30 days`, re-sends the branded `INVITATION` email. Audit `RESEND_INVITATION`. Allowed only on PENDING rows.
- **Revoke** — sets `status = EXPIRED`, soft-deletes the auth user if it was never accepted (`status = PENDING && authUserId is not null`). Audit `REVOKE_INVITATION`. Allowed only on PENDING rows.

Both actions:
- Live in `src/app/(admin)/invitations/actions.ts` as new server actions (`resendInvitationAction(invitationId)`, `revokeInvitationAction(invitationId)`).
- Restricted to ADMIN via `requireRole([Role.ADMIN])`.
- Wrapped in a small confirmation dialog (shadcn `AlertDialog`) on the client side.

ACCEPTED and EXPIRED rows show a static "—" in the actions column.

### 4.8 Audit trail

New audit actions written to `audit_logs`:
- `CREATE_INVITATION` — already exists, expand metadata to include `authUserId`.
- `RESEND_INVITATION` — new.
- `REVOKE_INVITATION` — new.
- `ACCEPT_INVITATION` — new (currently missing).

EntityType = `"Invitation"` for all four.

### 4.9 Backwards compatibility note

After this PR:
- Any unaccepted invitations sent with the legacy `?invitationId=…` link will **stop working** the moment the PR is deployed. The bursary office should resend them from the admin Invitations page (using the new Resend button) immediately after deployment.
- Accepted and Expired rows are unaffected.
- The seed data unchanged; existing tests should still pass.
