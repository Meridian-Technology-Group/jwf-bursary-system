## 1. Executive Verdict

**Recommendation: CONDITIONAL GO.** The picture has changed materially since the
17 May audit, which returned a HOLD against 14 of 32 Fail sections. Of the 15
original blockers, **12 are genuinely resolved and independently re-walked** —
all 81 walkthrough guides in `docs/guides/walkthroughs/` now pass cleanly (see the
companion VERIFICATION-PLAN). The system is no longer in a "can't submit / can't
open a round / can't complete an assessment" state. Feature acceptance under the
Foundation's verification checklist is now plausibly achievable.

**What remains is not feature work.** The outstanding items are security
hardening, production cutover/provisioning, contractual compliance activities,
and documentation deliverables. None require deep engineering, but several are
hard MSA gates that cannot be waived:

- **B10 virus scanning** is still entirely unimplemented (only a `TODO` comment).
- **B14 production domain / "live" deploy / Sentry** is not done.
- **B15 documentation** (user, admin, ops runbooks, hypercare) does not exist.
- A handful of **Tier-2 security one-liners** (email-merge XSS, CSV formula
  injection, rate-limiter fail-open, prod advisor sweep) remain open and will be
  flagged by the contractual pen test.

**Realistic path to acceptance.** ~1.5–2 weeks of focused engineering + ops,
plus external lead time for the independent pen test (clause 14.6) and the
ten-case parity test (Schedule 1 §3). The hard part — feature correctness — is
essentially done.

## 4. Remaining Go-Live blockers (verified open)

### 🔴 B10 — Virus scanning not implemented *(MSA Schedule 4 §8 + NF-10)*
The only trace in the codebase is a `TODO(B10)` comment in
`src/app/api/documents/[id]/url/route.ts`. There is no scanning step, no
`virusScanStatus` column, and no quarantine gate on URL issuance. Every document
an assessor opens is unscanned. The MSA explicitly commits to "virus scanning of
uploads," so this is both a day-one contract breach and a certain pen-test
finding.
**Fix:** add `Document.virusScanStatus` (`PENDING|CLEAN|INFECTED|FAILED`);
scan on upload via ClamAV (Supabase Edge Function / queue) or a scanning API for
MVP volume; refuse to sign URLs for non-`CLEAN` files. **Effort: M–L.**

### 🔴 B14 — No production domain, project not "live", no Sentry
- Vercel reports `"live": false`; latest deployment `target: null` (a preview,
  not a production deploy).
- No custom domain attached — only `*.vercel.app`. `NEXT_PUBLIC_APP_URL` and the
  Resend sender domain depend on a real domain.
- `@sentry/nextjs` is still absent from `package.json` — the contractually
  budgeted error monitoring does not exist.
**Fix:** register/attach the Foundation domain as a Production alias; populate
Vercel Production env per the matrix in the prior doc §10 (incl. verified Resend
domain and `KV_REST_API_*`); promote a real `main` production deployment; wire
Sentry with EU data residency. **Effort: M.**

### 🔴 B15 — No operational/user documentation *(MSA Schedule 1 §4)*
No `docs/ops/` directory exists; no user guide, admin/assessor guide, deployment
runbook, backup/restore procedure (with a logged drill), incident-response
playbook, or hypercare playbook.
**Fix:** author the Schedule 1 §4 deliverable set; execute and log one nonprod
restore drill. **Effort: L.**

### 🟠 Production is provisioned but not bootstrapped
Prod has **0 rounds, 0 profiles, 0 applications**. This is correct for a clean
environment, but it means **no ADMIN account exists in prod** and **no OPEN
round exists** — so on day one nobody can sign in as staff or invite an
applicant. This is a defined cutover task, not a defect, but it must be
sequenced into go-live.
**Fix:** bootstrap the first ADMIN; create + OPEN the 2026/27 round. **Effort: S.**

## 5. B8 MFA — built, with accepted gaps

MFA is implemented and enforces at `aal2` for staff roles in production
(`isStaffMfaEnforced()` defaults to true when `VERCEL_ENV === "production"`;
`STAFF_MFA_ENFORCED` overrides). Enrolment → aal2 → `/queue` (no redirect loop),
the challenge path, applicant pass-through, and the admin per-user Reset-MFA
control are all re-walked green (VERIFICATION-PLAN pass 9, admin/14).

Conscious go-live decisions still outstanding:
- **No backup/recovery codes.** A staff member who loses their authenticator is
  locked out until an admin runs Reset MFA — and the *first* admin cannot be
  reset by anyone. Recommend shipping backup codes before go-live.
- **Duplicate-factor race:** the first `/login/mfa` hit immediately after login
  can create a duplicate unverified factor ("factor … already exists"); a reload
  recovers. Noted in `docs/backlog/b8-mfa-for-admin-and-assessor.md`.
- **WebAuthn** deferred.
